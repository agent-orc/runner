The boring part that's secretly hard

Spawning a coding-agent CLI from another process and reading its output looks trivial — until you do it on Windows, unattended, at scale. A `.cmd` shim silently truncates your prompt. The agent commits to git when it shouldn't. Concurrent runs trample each other's session. A "done" marker fires because the agent read one in a file. CodingAgentRunner is the layer that has already paid for those lessons.

// run an agent, consume a structured event stream

What it hardens

Each of these was a real, time-consuming incident in a production orchestrator. Every fix ships with a test that pins why it exists.

Prompt delivery

Resolves the Windows `.cmd` shim to the real `.exe`, so a multi-line prompt isn't truncated at the first newline by cmd.exe.

Platform-owns-git

Keeps the agent from running git commit/push — the host owns version control — closing a git-bash bypass and a crash trigger.

Clean-context

Each run gets an isolated CLI home, so concurrent runs and your own session never collide.

Real completion

Completion comes from the CLI's own signal (the stream-json result frame + process exit), not a fragile [[TASK_DONE]]-scraping heuristic.

Lifecycle & reaping

Stop a run and reap the whole process tree — no orphaned grandchildren holding file handles.

Quota, cached

Probe the remaining rate-limit window per CLI, cached with an escalation policy that polls more often as you near the limit.
